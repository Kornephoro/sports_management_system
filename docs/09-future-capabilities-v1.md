# Future Capabilities v1

## 1. 文档目的

本文档用于记录当前已经明确有价值、但**不进入 v1 当前开发轮次**的未来能力方向。

本文档的目标是：

- 保留未来方向的项目记忆
- 让后续 agent 识别“这是已确认方向，但当前不应擅自实现”
- 为未来扩展提供统一追加入口
- 避免未来想法散落在聊天记录中

本文档不是：

- 当前 build order
- 当前 schema 设计文档
- 当前必须立即落地的功能清单

---

## 2. 使用方式

### 2.1 适合写进本文档的内容
只有以下类型的内容适合进入本文档：

- 已确认有价值的未来能力方向
- 当前不进入开发，但未来大概率会做的模块
- 需要项目持续记住的扩展方向
- 对底层模型有“未来留口”意义的想法

### 2.2 不适合写进本文档的内容
以下内容不要写入本文档：

- 当前轮次必须立即开发的功能
- 已经进入 build order 的任务
- 已经落地的功能说明
- 临时灵感但尚未确认价值的点子
- bug、技术债、当前实现备注（这些应进 08 文档）

### 2.3 进入本文档，不等于进入当前开发
本文档中的内容默认都属于：

- 已确认方向
- 当前不实现
- 当前不改 schema
- 当前不加入 build order

### 2.4 未来正式开发时的处理方式
当某个方向要正式进入开发时，应再补充到：

- `/docs/02-core-domain-v1.md`
- `/docs/04-architecture-v1.md`
- `/docs/06-build-order.md`
- 或新增专项文档

## 6. 能力条目：训练场景物品清单（Gym Carry Checklist）

### 6.1 需求背景
用户在去健身前经常忘记携带某些物品，例如：

- 训练腰带
- 护腕 / 护膝
- 鞋
- 水壶
- 毛巾
- 换洗衣物
- 补剂 / 小零食
- 耳机
- 门禁卡 / 健身卡

该能力的核心价值是：

- 降低忘带成本
- 提升训练前准备效率
- 让训练场景准备更系统化

### 6.2 当前定位
该方向当前建议归类为：

- 训练场景准备能力
- 未来可与 `Inventory / 物品柜` 模块联动
- 未来可与 `Purchase / Expense` 模块联动
- 未来可与 Session / Program 的训练场景需求关联

### 6.3 未来可能的扩展方向
未来该能力可能自然扩展为：

- `InventoryItem`
- `SessionChecklistTemplate`
- `SessionChecklistItem`
- Program / Session 级别的 `required_item_tags`
- 基于训练类型生成不同清单
- 基于用户历史遗漏生成提醒

### 6.4 当前处理原则
当前阶段：

- 该方向已确认有价值
- 当前不进入 v1 主链路
- 当前不新增正式实体
- 当前不修改 schema
- 当前只保留未来扩展认知

### 6.5 对底层认知的提醒
系统未来不应只理解“训练内容”，还可能需要理解：

- 训练前准备需求
- 训练场景所需物品
- 某些 Session 对装备的依赖关系

---

## 7. 能力条目：热量计算 / 基础代谢 / 减脂相关公式

### 7.1 需求背景
未来系统应支持与减脂、热量管理、代谢估算相关的能力，例如：

- 基础代谢（BMR）估算
- 总消耗（TDEE）估算
- 热量缺口计算
- 体重变化与热量平衡关系的推导
- 常见减脂经验规则与标准模板
- 与减脂目标相关的计算辅助

### 7.2 典型内容范围
未来可能涉及但当前不实现的内容包括：

- 常见 BMR 公式
- 热量缺口估算
- 某些减脂经验模板
- 与体重、体脂、腰围等身体指标联动的推导
- 结合饮食与体重变化做趋势判断

### 7.3 当前定位
该方向当前建议归类为：

- 身体数据输入的未来计算层
- 营养 / 代谢 / 减脂支持能力
- 未来的 `DerivedAssessment`、代谢计算器或相关模块候选

### 7.4 当前处理原则
当前阶段：

- 原始身体数据仍由 `Observation` 承载
- 代谢公式、热量计算结果不应直接混入 Observation 原始记录
- 当前不实现公式引擎
- 当前不扩展 Nutrition 系统
- 当前只在底层认知上保留“Observation 将来会作为代谢/减脂计算输入”的空间

### 7.5 对底层认知的提醒
未来应区分：

- 原始观测值
- 由公式推导出的计算值
- 基于计算值得出的建议

也就是说，未来系统不应把“输入”和“推导结果”混为一层。

- 当前核心主链路已经稳定
- build order 已推进到相关模块
- 该能力已成为高频真实痛点
- 已经明确需要对应的正式实体或计算模块
- 已经准备好单独的专项文档与实现切片

## 11. 当前结论

截至当前版本，以下未来方向已经被正式确认，但尚未进入实现：

1. 训练场景物品清单 / Gym Carry Checklist
2. 热量计算 / 基础代谢 / 减脂相关公式支持

它们应被视为：

- 产品未来能力的一部分
- 当前开发阶段的留口方向
- 后续需要单独设计和切片推进的扩展模块

## 12. 能力条目：一分化渐进超负荷自动化与变化高亮

### 12.1 需求背景
当前用户在 Excel 中已经形成了一套一分化训练的实际使用方法，其核心不是“每天完全重做计划”，而是：

- 大部分动作保持不变
- 每次只选择 1~2 个动作发生进步
- 进步通常体现为：
  - 增加组数
  - 或达到阈值后增加重量
- 某些动作不适用该进步方式（例如俯卧撑、平板支撑）

同时，用户非常重视类似 Excel 的“高亮交互感”，即能够快速看出：

- 哪个动作变了
- 变的是哪一个字段
- 为什么发生变化

### 12.2 当前定位
该方向当前建议归类为：

- 训练主链上的未来强化能力
- 计划生成与执行展示之间的桥接能力
- 训练自动化与高亮提示能力

### 12.3 未来希望实现的能力
未来应支持至少以下能力：

1. 一分化训练的最小渐进超负荷自动化  
2. 基于动作进步轨道的变化判定  
3. 对“总次数攒够 n 次即进阶”的动作做自动推进  
4. 对不适用该规则的动作区别处理  
5. 在 planned sessions / today / execute 等关键页面高亮提示：
   - 哪个动作变化了
   - 变的是组数 / 次数 / 重量 / RPE / 时长
   - 变化原因是什么

### 12.4 当前处理原则
当前阶段：

- 该方向已确认有价值
- 当前不要求立刻进入小轮次修复
- 当前不应在硬阻塞修复轮中顺手实现
- 后续应作为训练主链的一个独立切片推进

### 12.5 对底层认知的提醒
未来系统不应只理解“当前计划是什么”，还应逐步理解：

- 某个动作为什么进步
- 当前进步处于哪个阶段
- 哪些动作本次应该变化
- 变化应如何被用户快速感知

### 12.6 当前结论
“一分化渐进超负荷自动化与变化高亮”已经被确认是训练主链中的高价值能力。  
但它应在当前硬阻塞修复完成后，作为单独切片开发，而不是继续混在当前小修轮次中推进。


## 19. 训练复盘与干预层（后续高优先级方向）

### 19.1 定位
当训练主链做到以下能力后：

- 建模板
- 生成安排
- 执行录入
- 自动进步
- 变化高亮
- 例外后重算
- 减载建议

下一步最值得做的大块，不应再是单纯增加更多录入字段，而应是：

> 训练复盘与干预层

也可以理解为：
- 训练复盘中心
- 瓶颈分析中心
- 干预建议中心

### 19.2 作用
这个模块的价值不在于“再做一个 dashboard”，而在于回答：

- 这周 / 这阶段练得怎么样
- 哪些动作在推进
- 哪些动作卡住了
- 哪些动作经常 partial / skipped
- 哪些动作接近减载窗口
- 哪些动作已经进入实现轮 / 停滞期
- 下一步最值得干预的地方在哪里

### 19.3 为什么它重要
训练系统如果只有：
- 计划生成
- 执行记录
- 自动变化

但没有一个帮助用户理解这些结果的复盘层，那么数据仍然是散的。  
复盘层的作用是：

> 把训练过程中的“变化、停滞、异常、恢复、减载”聚合成可理解的训练判断。

### 19.4 建议包含的内容
#### A. 本周 / 本阶段训练完成情况
- 完成次数
- skipped / partial 情况
- 哪些训练被补练或重排

#### B. 动作级推进状态
- 最近若干次暴露的推进情况
- 常规推进次数
- 实现轮次数
- 保持次数
- 计划减量次数
- 异常调整次数

#### C. 停滞与瓶颈提示
- 哪些动作长时间未推进
- 哪些动作多次未达标
- 哪些动作连续 partial / skipped
- 哪些动作应考虑替换或减载

#### D. 恢复与干预建议
- 疲劳状态趋势
- pain / discomfort 聚合
- 是否建议减载
- 是否建议局部动作替换
- 是否建议调整未来安排

#### E. 下一步关注重点
- 下一次训练重点动作
- 即将实现进阶的动作
- 下次可能进入减载或恢复的动作

### 19.5 页面定位建议
这个模块未来不应只做成“统计页”，而应更偏：

> 训练决策辅助页

它的重点是帮助用户判断：
- 现在发生了什么
- 接下来该关注哪里
- 是否需要干预

### 19.6 当前阶段建议
该模块建议放在训练主链完成稳定自用后再推进。  
优先级高于：
- 大型 dashboard
- AI 推荐
- 非主链扩展功能

因为它直接提升训练系统的“理解力”和“决策价值”。
