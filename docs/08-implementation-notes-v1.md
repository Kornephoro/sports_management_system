- 某些 API 通过 `userId` query 参数传递用户身份

- 真实登录态
- 不再在 query 参数中长期传递 `userId`

这是一个明确的技术债，但当前不是阻塞项。

## 13. Evidence 实现提醒

### 13.1 当前 parse 仍为 mock
当前 Evidence 模块已经完成最小闭环：

- upload
- `EvidenceAsset(parse_status=pending)`
- mock parse
- confirm / reject

但当前解析链路仍然是 **mock parse**，不是真实 AI / vision 解析。

### 13.2 当前状态可以接受
这符合第 7 轮“最小可验证闭环”的目标。  
当前的重点是验证：

- 文件上传链路
- EvidenceAsset 状态流转
- confirm / reject 交互
- confirm 后的最小数据落点

### 13.3 后续必须持续明确标注为 mock
后续无论在：

- 页面
- 文档
- agent prompt
- 开发说明

都应持续明确当前 parse 为 mock，避免后续误判为“真实解析能力已经完成”。

### 13.4 Confirm 当前只落到 Observation
当前 `confirmParsedEvidence` 的最小落点选择为 `Observation`。  
这是 v1 阶段的最小实现决定，原因是：

- 最容易验证
- 与当前 Observation 闭环能快速联动
- 不会把 EvidenceAsset 与 Execution / Observation 混成单一对象

### 13.5 后续注意
当前 `confirm -> Observation` 不应被误认为未来唯一落点。  
后续不同类型的 evidence 很可能需要分流到不同目标，例如：

- 训练截图 -> SessionExecution / UnitExecution
- 营养截图 -> Nutrition / Observation / 后续计算层
- 伤病相关证据 -> InjuryIncident / ConstraintProfile
- 身体数据截图 -> Observation

### 13.6 当前结论
Evidence 模块当前已经满足 v1 的最小闭环验证要求，但后续扩展时必须重新设计：

- parse 类型分流
- confirm 落点策略
- mock parse 到真实 parse 的替换路径


## 14. Constraint / Injury 实现提醒

### 14.1 当前 Constraint / Injury 只做到最小闭环
当前第 8 轮已经实现：

- create constraint
- list active constraints
- resolve constraint
- create injury
- list injuries
- link injury to constraint
- 计划生成时写入 `constraint_snapshot`

这已经足够验证“Constraint / Injury 已进入系统”，但不代表康复与限制系统已经成熟。

### 14.2 当前 constraint-aware planning 只是轻量影响
当前计划生成链路中对 active constraints 的影响仍然是最小版：

- 读取 active constraints
- 做最小标签交集判断
- 写 `constraint_snapshot`
- 标记 `affected = true`
- 写轻量 warning（如 `constraint_affected_unit`）

当前没有实现：

- 自动替换动作
- 自动删减组数
- 自动调低疲劳
- 自动重排 session
- rehab 驱动的动态调度

### 14.3 当前匹配逻辑应被视为开发期最小规则
当前匹配逻辑使用的是最小标签交集：

- `progress_track_key`
- `movement_pattern_tags`
- `contraindication_tags`
- `fatigue_tags`
- constraint 的 `movement_tags`
- `body_region_tags`
- `restriction_rules` 中的部分字段

这一版逻辑的目标是验证“约束信息能否进入计划链路”，不是为了追求高准确度或完整性。

### 14.4 Injury 与 Constraint 当前关系仍然很轻
当前实现中：

- InjuryIncident 可以独立存在
- ConstraintProfile 可以链接一个 InjuryIncident
- 这种关系当前主要服务于“最小结构化关联”

当前未实现：

- 一对多 / 多对多的伤病-约束关系管理
- 自动从 Injury 生成 Constraint
- 伤病恢复阶段建模
- return-to-training 评分或路径建议

### 14.5 当前页面仍然是开发态验证页面
`/constraints` 页面当前主要服务于：

- 录入
- 查看
- resolve
- link
- 验证计划链路已接入 constraint-aware

它还不是完整的康复管理工作台。

### 14.6 当前结论
Constraint / Injury 现在已经足以继续推进项目，但后续若要扩展康复能力，必须单独设计：

- 更成熟的 constraint-aware 调整策略
- rehab 计划与 rehab 单元
- injury 生命周期与恢复阶段
- 更明确的页面和规则边界

## 18. Program 主链路可用性修复提醒

### 18.1 第一次内部演示阻塞的真实根因
第一次内部演示中，Planned Sessions 生成失败的主要根因并不是 Demo Program 本身错误，而是：

- 回归脚本持续创建了多个最小 Program
- 这些 Program 缺少 block / session template 结构
- 列表按创建时间排序时，它们容易排在最上方
- 演示时被优先点开，造成“主链路看起来坏了”的错觉

### 18.2 当前修复策略
第 12 轮采取的修复策略是最小可用性修复，而不是重做主链路：

- Program 列表增加 planning readiness
- 首页增加 Demo Program 引导
- Planned Sessions 页面增加生成前就绪判断
- 不可生成时直接提示并禁用按钮
- Program 详情页增加模板层级术语说明

### 18.3 当前实现边界
这轮修复的是：

- 主链路防误选
- 术语理解成本
- 页面最小 loading / 提示

当前没有做：

- 大规模性能优化
- 复杂主链路向导
- 完整 onboarding
- 全局中文化
- 全面 UX 重构

### 18.4 当前结论
第 12 轮之后，Program 主链路已从“容易因误选而阻塞”进入“有最小引导、最小保护、最小术语解释”的状态。  
这足以支持下一次内部演示优先复验主链路是否真正变顺。

- 清理无关 planned sessions / program 噪音
- 校正“模板 -> 生成 -> 执行 -> 按动作核销 -> 聚合状态”的主链方向
- 修复“录了 SessionExecution 后找不到继续编辑入口”
- 修复动作列表异常缺失问题

### 33.2 当前清理策略提醒
当前系统已经增加“清理无用实验训练数据”的能力。  
其目标不是删除一切数据，而是：

- 保留真实执行记录
- 删除回归 / 演示噪音数据
- 让 Planned Sessions 页面更接近真实使用环境

### 33.3 当前主链收口提醒
当前训练主链的正确理解应是：

1. 模板定义训练内容
2. 生成若干次已安排训练
3. 进入执行录入
4. 先录整次训练信息
5. 再按动作逐项核销
6. 再由系统聚合状态

后续任何增强，都应继续围绕这条链，而不是重新发散。

### 33.4 当前恢复编辑提醒
当前系统已经开始支持“按 PlannedSession 找回最近执行记录”。  
这意味着：
- 填完基本信息后，记录不应再给用户“消失了”的感觉
- 用户应能从安排页 / today / 训练记录页继续回到这条链

### 33.5 当前结论
该轮次完成后，训练主链已从“可演示”进一步推进到“更接近真实自用”，但仍需继续补上：
- 把训练拉回今日 / 下一次训练
- 在训练计划详情中调整动作
这两项完成后，训练主链会更接近稳定自用。

## 34. 训练执行核销逻辑 v1

### 34.1 目标
训练执行记录的目标不是把每组每次都强制详细录入，而是：

- 用最少填写成本
- 准确记录与原计划相比的关键偏差
- 为后续训练结论与进步判断保留足够信息

### 34.2 三层结构
当前训练执行核销逻辑 v1 分为三层：

#### A. 整次训练总结（SessionExecution）
至少包括：
- 整体感受：轻松 / 正常 / 困难
- 实际时长
- 备注

#### B. 动作级核销（UnitExecution）
每个动作只使用 3 个主状态：
- 已完成
- 部分完成
- 未做

#### C. 偏差记录
动作主状态之外，允许记录与计划相比的偏差，例如：
- 少做组数
- 少做次数
- 加重
- 降重
- 加组
- 加次数
- 替换动作
- 执行方式变化
- 备注

### 34.3 原因标签
偏差可附带原因标签，至少包括：
- 力竭
- 时间不够
- 状态差
- 疼痛 / 不适
- 器械限制
- 场地限制
- 其他

### 34.4 判定规则
#### 已完成
满足以下情况之一时可判为已完成：
- 按计划完成
- 比计划更强地完成（加重 / 加组 / 加次数）
- 虽有执行方式变化，但核心工作量完成
- 轻微降重但仍完成全部目标组数与基本目标次数

#### 部分完成
满足以下情况之一时可判为部分完成：
- 少做组数
- 某组明显未达到目标次数
- 动作只完成一部分
- 替换动作且完成度明显下降
- 原计划内容未被基本完成

#### 未做
- 该动作完全未做

### 34.5 关键例子
#### 例 1
计划 3 组，只完成 2 组  
=> 部分完成 + 少做组数

#### 例 2
计划 3 组，最后一组降重但仍完成  
=> 已完成 + 降重

#### 例 3
计划最后一组 8 次，实际只做了 5 次力竭  
=> 部分完成 + 少做次数 + 力竭

#### 例 4
两个动作改成超级组，但都完成  
=> 已完成 + 执行方式变化（超级组）

#### 例 5
计划太轻松，实际增加重量 / 次数 / 组数  
=> 已完成 + 加重 / 加次数 / 加组

#### 例 6
动作被替换为其他动作  
=> 主状态按完成度判断，偏差记录为替换动作

#### 例 7
时长类动作（如平板支撑）  
=> 按时长核销，不强套普通次数逻辑

### 34.6 交互原则
当前交互原则为：

> 默认按计划完成，只记录偏差

也就是说：
- 每个动作先展示计划值
- 用户先只选择主状态
- 只有当存在偏差时，才展开最小填写区

### 34.7 自动聚合整次训练结论
整次训练结论至少包括：
- 已完成
- 部分完成
- 未训练

当前聚合规则为：
- 全部动作基本完成 => 已完成
- 存在部分完成或未做 => 部分完成
- 整次未开始 => 未训练

### 34.8 当前结论
这套 v1 规则的目标不是做完整训练日志系统，而是先让训练执行记录：
- 足够准确
- 足够低负担
- 足够支持后续进步逻辑

后续若真实使用中发现某些高频情况不足，再单独迭代。

## 36. 核销卡片的原计划基准、条件显示与互斥约束补充

### 36.1 原计划基准原则
训练执行核销的核心前提是：

> 所有填写都应基于预先安排的计划

因此，核销卡片中必须清楚显示原计划摘要，包括：
- 计划组数
- 计划次数 / 时长
- 计划重量
- 如适用，计划 RPE 范围

### 36.2 组级预填原则
若用户进入组级明细，不应从空白开始填写。  
系统应按原计划自动预填每一组的默认值，用户只修改发生偏差的地方。

目标是：
- 减少重复输入
- 让核销更像“校对 / 修正”而不是“重录”

### 36.3 条件显示原则
核销卡片不应把所有输入框同时摊开。  
应分为：

#### 通用输入区
始终显示的内容，例如：
- 主状态
- 原计划摘要
- RPE
- 疼痛评分
- 备注

#### 条件输入区
仅在选择对应偏差后显示的内容，例如：
- 少做组数 => 显示实际完成组数
- 少做次数 => 显示实际完成次数
- 加组 => 显示加组数量
- 加次数 => 显示加次数数量
- 替换动作 => 显示替换动作名称
- 执行方式变化 => 显示执行方式类型与说明

### 36.4 互斥约束原则
为避免逻辑冲突，偏差标签至少应实现以下互斥：

- 少做组数 vs 加组
- 少做次数 vs 加次数
- 加重 vs 降重

同时，当主状态 = 未做 时，应限制无关偏差输入，仅保留原因和备注。

### 36.5 设计目标
该补充的目标是让核销卡片从“通用表单”进一步收口为：

> 基于原计划的偏差记录器

既能表达真实训练中的复杂情况，又不显著增加填写负担。

## 37. 核销卡片交互优化补充（已确认方向）

### 37.1 当前确认采纳的优化方向
在现有“训练执行核销逻辑 v1”基础上，以下交互优化方向已确认可进入后续实现范围：

#### A. 主状态需要文字解释，不只靠颜色
“已完成 / 部分完成 / 未做”不应只依赖按钮颜色区分。  
在用户选中后，应给出一句简短解释，帮助用户快速确认自己是否点对，例如：

- 已完成：按计划完成，或虽有调整但整体完成
- 部分完成：有组数 / 次数 / 重量 / 时长等未按计划完成
- 未做：该动作本次未执行

目标是降低误判与二次确认成本。

#### B. 部分完成后的默认引导，不要求用户从空白开始填
当用户选择“部分完成”后，系统不应只给出一堆空白字段，而应根据最常见偏差路径提供默认引导。

当前已确认的方向是：
- 支持“基于偏差标签触发默认编辑逻辑”
- 例如：点击“少次数”后，可默认把最后一组的次数留空或置为待填写状态
- 用户只需补上实际次数，而不是重填整张卡

这类逻辑本质上是“默认按计划完成，只对最可能发生偏差的最后一组做快捷处理”。

#### C. 原计划与实际值应并排展示
核销卡片不应只展示抽象输入框，而应尽量让用户直观看到：

- 计划值
- 实际值

理想方向例如：
- 计划：60kg × 8
- 实际：55kg × 5

目标是让用户不需要在脑中自行换算“到底和原计划差了什么”。

#### D. 偏差标签除互斥外，还应尽量受动作类型约束
除基础互斥外，后续应进一步考虑“偏差标签与动作类型的适配关系”，避免出现无意义组合。

例如：
- 时长类动作不应默认出现“少做次数”这种不自然表达
- 未做状态下不应继续暴露大量与完成动作相关的偏差项
- 替换动作时，应优先聚焦“替换成了什么”，而不是同时堆出一大批无关偏差字段

目标是进一步减少逻辑冲突和填写噪音。

#### E. 次数类动作与时长类动作应区别呈现
后续核销卡片应考虑根据动作类型切换呈现方式：

- 次数型动作：组数 / 次数 / 重量
- 时长型动作：组数 / 时长 / 负重（如适用）

避免把时长类动作强行塞进次数型表单逻辑，降低理解成本。

#### F. 原因标签未来可区分为“训练内原因 / 外部原因”
当前原因标签已可用，但后续可考虑在不增加明显填写负担的前提下，将原因标签进一步整理为两类：

- 训练内原因：力竭、状态差、疼痛 / 不适
- 外部原因：时间不够、器械限制、场地限制

这样做的价值在于：
- 更利于后续分析训练表现
- 更利于后续进步逻辑做不同处理

该方向当前作为已确认的后续优化点保留。

### 37.2 当前明确不采纳的方向
当前阶段，以下两项不作为既定方向写入实现要求：

#### A. 不强制修改“收起调整”等入口命名
该项当前不作为明确优化目标冻结，后续如真实使用中继续造成理解问题，再单独处理。

#### B. 不单独增加“只调整最后一组”的独立快捷入口
当前不要求增加一个单独按钮型入口。  
后续更倾向于通过“偏差标签触发默认编辑逻辑”来覆盖这一需求。

例如：
- 点击“少次数”后，默认把最后一组的次数留空等待填写
- 点击“降重”后，默认把最后一组重量切换到可编辑状态

这种方式能覆盖大量“最后一组发生偏差”的场景，同时减少额外按钮和交互复杂度。

### 37.3 当前结论
上述方向的共同目标是让核销卡片进一步收口为：

> 基于原计划的偏差记录器

也就是说：
- 默认按计划完成
- 只在有偏差时让用户补充最少必要信息
- 让真实训练中“最后一组出问题”这类高频情况更自然可填
- 继续避免系统演化成高负担的完整逐组训练日志工具

## 38. 单次训练动作调整页与结构化负重提醒

### 38.1 本轮定位
本轮不是扩业务模块，而是对训练主链中的“单次训练层动作查看与调整”做最小收口。

### 38.2 当前统一入口
系统当前已收口为一个统一的“查看动作清单与调整”页面：
- `/programs/[programId]/planned-sessions/[plannedSessionId]/plan`

这意味着：
- `/today` 和 Planned Sessions 不再各自维护不同调整入口
- 单次训练层动作调整开始有统一落点

### 38.3 当前负重建模方向
本轮明确把动作负重从字符串形式推进到结构化方向：

- 普通负重动作：`load_value + load_unit`
- 自重动作：`bodyweight_plus_external`

同时兼容保留 `load_text` 作为展示派生值，但后续逻辑应优先使用结构化字段。

### 38.4 当前自重动作方向
当前自重动作（例如俯卧撑）已开始走正确方向：

- 自重来源优先联动 observations 最新 bodyweight
- 附重单独记录
- 后续容量统计与进步逻辑应基于“自重 + 附重”理解，而不是把自重动作硬塞进普通负重逻辑

### 38.5 当前技术债提醒
本轮结束时，`verify:v1` 仍存在非本轮直接引入但必须记录的问题：

- `Program_goal_id_fkey` 外键约束失败
- 说明当前数据 / verify 基线仍有不一致

该问题虽然不阻断动作库主线继续推进，但必须在后续某轮补回基础稳定性。

### 38.6 当前结论
本轮完成后，训练主链已经具备：
- 单次训练动作调整统一入口
- 结构化负重方向
- 自重动作的正确建模方向

这为后续：
1. 动作库
2. 模板库
3. 动作进步逻辑

提供了合适的承接基础。

## 34. 动作库首页

### 34.1 页面定位
动作库首页是“动作总览与管理页”，不是训练页、模板页或执行页。

它的职责是：
- 浏览系统中已有动作
- 搜索 / 筛选动作
- 新建动作
- 停用 / 归档动作
- 进入某个动作详情页

### 34.2 页面形态
首页更适合采用动作卡片总览，而不是纯表格或大段表单。  
每张动作卡片只承载最小摘要信息，例如：

- 动作名称
- 记录方式
- 负重模型
- 活跃状态
- 最近使用
- 快捷入口：查看详情 / 编辑 / 停用

首页不应承载过多动作定义细节、历史分析或完整肌群配置。

### 34.3 首页目标
动作库首页的价值不是“展示很多字段”，而是让用户快速完成：

- 找动作
- 新建动作
- 进入动作详情
- 把动作用于安排训练

## 35. 动作详情页

### 35.1 页面定位
动作详情页不是简单的“编辑动作表单”，而是动作的定义与使用信息页。

动作详情页服务于训练安排，但不应被设计成孤立百科页。

### 35.2 主要信息分层
动作详情页建议分为四层：

#### A. 动作定义层
回答“这个动作是什么”，包括：
- 标准名称
- 别名
- 记录方式
- 负重模型
- 默认安排参数
- 动作模式
- 主要训练部位 / 次要训练部位
- 动作做法
- 注意事项
- 拉伸 / 准备建议

#### B. 动作档案摘要层
回答“这个动作最近练得怎么样”，包括：
- 历史训练次数
- 最近一次训练
- 最近表现摘要
- 最佳重量
- 最佳次数
- 最近趋势

#### C. 引用关系层
回答“这个动作当前在哪里被使用”，包括：
- 模板引用
- 计划引用
- 最近训练引用

#### D. 治理层
回答“这个动作是否干净、统一、可维护”，包括：
- 活跃 / 归档状态
- 疑似重复动作
- 别名管理
- 合并入口

### 35.3 全局跳转原则
动作详情页应成为系统中“动作”的统一落点。

理想状态下，计划页、单次训练调整页、已安排训练页、execute 页、训练记录页中出现的动作名称，只要存在明确动作映射，都应能跳到同一个动作详情页。

## 36. 肌群选择与人体图联动

### 36.1 交互目标
动作详情页中的“主要训练部位 / 次要训练部位”不应只是文本标签，而应尽量具备可视化联动。

### 36.2 左右结构
建议采用：
- 左侧：主要训练部位 / 次要训练部位标签选择区
- 右侧：人体图示意区

### 36.3 联动规则
- 选择主要训练部位时，对应区域在人体图中以深色高亮
- 选择次要训练部位时，对应区域以浅色高亮
- 后续若交互允许，人体图点击也可反向同步到标签选择区

### 36.4 当前颗粒度
当前建议使用“区域级”颗粒度，不做更细解剖颗粒度，以避免冗余和维护负担。

## 37. 通用进步逻辑框架（兼容不同训练安排）

### 37.1 目标修正
进步逻辑不应只服务于一分化，也不应被设计为“一套固定加重脚本”。  
更准确的目标应为：

> 构建一个兼容常规训练模式的通用进步框架，使不同计划、不同动作角色、不同进步策略都能在统一主链中运行。

这里的“常规训练模式”至少包括：
- 一分化 / 全身重复型
- 上下肢
- 推拉腿
- AB 轮转
- 固定周几训练
- preserve order / preserve calendar / smart merge / manual 等恢复策略下的训练顺延与保序

### 37.2 设计原则
当前建议采用以下原则：

1. **分化决定训练结构，不决定进步算法**
2. **进步规则挂在模板动作槽位上，不直接挂在动作名上**
3. **动作轨（ProgressTrack）保存运行时状态**
4. **执行结果驱动进步状态更新**
5. **下一次训练安排由模板配置 + 当前动作轨状态共同合成**
6. **变化结果必须天然支持页面高亮，而不是靠前端临时猜测**

### 37.3 分层模型
#### A. 动作库层
动作库只提供默认建议，不保存运行时进步状态。  
可承载例如：
- 默认记录方式
- 默认负重模型
- 默认推荐进步家族
- 默认动作模式 / 肌群
- 默认角色倾向（更像主项还是辅助项）

#### B. 模板动作槽位层
模板中的每个动作槽位（TrainingUnitTemplate）是进步配置中心。  
这一层应作为“该动作在该计划中的进步语义”承载位置。

应重点利用当前已存在的字段：
- `unit_role`
- `progression_family`
- `progression_policy_type`
- `progression_policy_config`
- `adjustment_policy_type`
- `adjustment_policy_config`
- `success_criteria`
- `progress_track_key`

#### C. 动作轨层（ProgressTrack）
`ProgressTrack` 是运行时状态中心。  
它不等于动作库动作，也不等于某次训练实例，而是：

> 某个模板动作槽位在连续多次训练中的持久进步轨迹

当前 schema 已明确：
- `(user_id, track_key)` 作为稳定身份
- `current_state` / `progression_policy_config` 用于承接运行时状态
- `exposure_count` / `success_count` / `failure_count` / `progression_count`
- `last_exposure_at` / `last_success_at` / `last_failure_at` / `last_progression_at`

#### D. 计划实例层
`PlannedUnit.progression_snapshot` 用于冻结“本次计划是如何根据进步逻辑生成出来的”。  
这样后续模板继续变化时，已生成的训练仍保留其生成语义。

#### E. 执行层
`UnitExecution.actual_payload` / `set_logs` / `result_flags` 用于承接真实执行结果。  
进步引擎未来应读取标准化后的执行结果，而不是依赖自由文本备注。

### 37.4 不同训练安排的兼容方式
#### A. 一分化 / 全身重复型
一分化只是该框架的一个实例。  
其典型逻辑可以是：
- 大部分动作保持不变
- 每次只选择 1~2 个动作发生进步
- 辅助项进入轮转池
- 达标后加重，或先加组再加重
- 页面明确高亮本次变化项

#### B. 上下肢 / AB 轮转
不同 session template 内的动作槽位拥有各自的进步轨。  
同名动作是否共轨，不应默认假设，后续如需要共享，应另设共享机制。

#### C. 推拉腿 / 多模板分化
不同模板中的主项、辅项、辅助项可绑定不同进步策略。  
引擎不需要知道“这是 PPL 还是上下肢”，只需要知道：
- 这次暴露了哪些动作槽位
- 每个槽位角色是什么
- 当前绑定的策略是什么
- 当前动作轨状态是什么

#### D. 固定周几 / 有序轮转 / 灵活窗口
调度与保序由 scheduling / recovery policy 负责。  
进步引擎只判断“这条轨道是否发生有效暴露”和“本次结果如何”，不把具体日历结构硬编码进进步逻辑。

### 37.5 角色层：主项 / 次主项 / 辅助项
#### A. 主项（main）
主项特征：
- 训练地位高
- 通常有稳定推进节奏
- 更适合脚本式、线性式或周期式推进
- 通常不进入“每次随机挑 1~2 个推进”的普通轮转池

#### B. 次主项（secondary）
次主项介于主项和辅助项之间：
- 仍需明确推进
- 但节奏可弱于主项
- 可采用双进阶、线性步进、曝光式推进等

#### C. 辅助项（accessory）
辅助项特征：
- 更偏容量、肌肥大、补短板
- 更适合阈值型、轮转型、加组型、总次数累积型推进
- 更适合进入“本次只挑几个动作变化”的候选池

#### D. 其他角色
`skill / conditioning / warmup / cooldown / mobility / prehab`  
通常不应强行纳入同一自动进步逻辑，应允许：
- 手动
- 保持
- 轻量 performance / duration 类型推进

### 37.6 通用策略体系建议
#### A. progression_family 只保留大类
建议继续使用当前 schema 中较抽象的家族层：
- `strict_load`
- `threshold`
- `exposure`
- `performance`
- `autoregulated`

#### B. progression_policy_type 承载具体常见策略
建议后续正式支持的策略类型至少包括：

##### 主项常用
- `linear_load_step`
- `linear_periodization_step`
- `scripted_cycle`

##### 辅项 / 辅助项常用
- `double_progression`
- `total_reps_threshold`
- `add_set_then_load`
- `reps_then_external_load`

##### 时长 / 自重 / 技术类
- `duration_threshold`
- `bodyweight_reps_progression`
- `hold_or_manual`
- `manual`

### 37.7 策略语义说明
#### A. linear_load_step
适合较直接的线性推进主项。  
完成当前条件后，按固定步长加重或提高目标值。

#### B. linear_periodization_step
适合按预设阶段推进的主项。  
重点不是“每次都变”，而是“按阶段切换目标”。

#### C. scripted_cycle
适合脚本式周期（例如后续可承接更完整的 5/3/1、周计划脚本等）。  
应由 `progression_policy_config` 明确每一曝光或每一周处于哪个阶段。

#### D. double_progression
适合常规辅项。  
当当前重量下的目标次数区间完成度达到标准后，再提升重量。

#### E. total_reps_threshold
适合“总次数攒够阈值再进阶”的模式。  
例如在当前重量下，本次或若干次暴露达到规定总次数后，再加重或升级。

#### F. add_set_then_load
适合你已经验证过的一分化式轮转推进：
- 先在当前重量上增加组数
- 达到阈值后再加重
- 加重后回到基础组数状态

#### G. reps_then_external_load
适合部分自重动作：
- 先提升次数或总次数
- 达到标准后再增加外加重量 / 附重

#### H. duration_threshold
适合时长动作：
- 先提升单组或总时长
- 达标后提升目标时长或难度等级

#### I. hold_or_manual / manual
适合不应自动推进的动作：
- 热身
- 康复
- 技术动作
- 暂不交给系统自动判断的动作

### 37.8 成功判定与执行输入
执行层已采用“默认按计划完成，只记录偏差”的逻辑。  
因此进步引擎未来不应直接看“是否提交过记录”，而应看标准化后的执行结果。

#### A. 进步引擎应至少读取
- 主状态：已完成 / 部分完成 / 未做
- 偏差标签：少组 / 少次数 / 加重 / 降重 / 加组 / 加次数 / 替换动作 / 执行方式变化
- 实际完成组数
- 实际次数 / 实际时长
- 实际负重 / 实际附重
- 可选的 RPE / pain / notes
- 若存在：`set_logs` 与 `actual_payload`

#### B. 建议归纳出的四类运行结果
1. **成功并达标**  
   已完成，且满足当前 `success_criteria`

2. **成功但未达标**  
   已完成，但还未达到进阶条件

3. **部分完成**  
   发生少组、少次数、明显下降完成度等

4. **未做**  
   本次该动作未执行

### 37.9 选择器：谁在本次应该进步
该系统不应默认“每次所有动作一起进步”，而应允许存在一个“本次谁变化”的选择器。

#### A. 主项
主项通常不进入普通轮转池，而是按自己的 script / line / period 走。

#### B. 次主项
可进入较小候选池，但优先级高于 accessory。

#### C. 辅助项
进入轮转池后，可基于以下因素选择：
- 距离上次进步时间
- 最近是否刚失败
- 主要肌群是否与已选动作冲突
- 动作模式是否冲突
- 当前 fatigue / constraint 信息
- 当前 session 中是否适合承担变化

#### D. 结果
这样可同时兼容：
- 一分化时每次只挑 1~2 个动作推进
- 多模板分化时在各自 session 内局部推进
- 主项与辅项采用不同推进节奏

### 37.10 变化高亮机制
变化高亮不应由前端临时对比旧值推断，而应由进步引擎在生成计划时直接产出结构化结果。

建议在 `progression_snapshot` 中包含：
- `before`
- `after`
- `changed_fields`
- `change_reason`
- `track_phase`
- `policy_type`

这样后续页面（如 Planned Sessions / Today / Execute）可以直接高亮：
- 哪个动作变了
- 变的是组数 / 次数 / 负重 / 时长哪一项
- 为什么变

### 37.11 v1 / v1.5 范围建议
#### A. v1 建议先支持
- `linear_load_step`
- `double_progression`
- `total_reps_threshold`
- `add_set_then_load`
- `bodyweight_reps_progression`
- `duration_threshold`
- `manual`
- `scripted_cycle` 最小框架预留

#### B. v1.5 再扩展
- 更完整的线性周期
- 更完整的 5/3/1 具体脚本
- autoregulated deload / retry / hold
- 多模板共享进步轨
- 失败恢复曲线
- 更深度的 constraint / fatigue 接入

### 37.12 当前结论
通用进步逻辑的推荐方向应为：

> 以模板动作槽位为配置中心，以 ProgressTrack 为运行状态中心，以执行结果为驱动输入，以计划生成时的 progression_snapshot 为输出快照的通用框架。

这样系统就不只支持一分化，而是天然兼容：
- 一分化
- 上下肢
- 推拉腿
- 固定周几
- 有序轮转
- 线性进步
- 双进阶
- 线性周期
- 主项 / 辅项差异
- 自重 / 时长 / 手动动作

### 37.13 当前阶段提醒
当前这套方案仍然属于“设计收口阶段”，不是立刻一次性实现全部策略。  
后续正式进入开发时，应继续坚持：

- 先收口统一框架
- 再按 build order 分批接入策略
- 避免为了支持某个单一训练法而把整个进步逻辑写死

## 39. 混合型周安排与非规律分化的兼容原则

### 39.1 目标澄清
通用进步逻辑不应只兼容“一分化、二分化、三分化”这些容易命名的分化类型。  
它还必须兼容如下更混合的训练安排，例如：

- 腿、胸、背、休、上半身、下半身、休
- 固定周结构中混合单部位日与组合日
- 不完全对称的周安排
- 有休息日插入的轮转结构

### 39.2 当前结论
这类混合型周安排在当前设计方向下应被视为：

> 可以兼容的训练结构，而不是特殊例外

原因不是系统“识别了四分化或五分化名称”，而是因为当前正确的设计方向应为：

- 调度层管理训练顺序与恢复策略
- 模板动作槽位管理进步配置
- ProgressTrack 管理运行时状态
- 执行结果驱动状态更新

### 39.3 为什么这类安排可以兼容
#### A. 分化名称不是核心
“腿、胸、背、休、上半身、下半身、休”这种结构，本质上只是一个 microcycle 中的 session 顺序安排。  
系统不应依赖“这是几分化”来决定进步逻辑。

#### B. 有效暴露才是核心
同一个动作的进步，不应主要按“自然周历”判断，而应优先按：

- 该动作轨的有效暴露次数
- 该动作轨最近的成功 / 失败 / 进步状态
- 当前绑定的策略配置

也就是说，某个动作一周出现一次还是两次，不会改变框架本身，只会影响该动作轨暴露节奏。

#### C. 同一个周结构中可以有不同 session 类型
在混合型周安排中，可能同时存在：
- 单部位日（如胸、背、腿）
- 组合日（如上半身、下半身）
- 休息日

这不应导致进步逻辑分裂。  
系统应只关注：
- 当前是哪一个 session template
- 暴露到了哪些动作槽位
- 这些槽位各自绑定了什么角色与策略
- 它们对应的 ProgressTrack 当前状态是什么

### 39.4 当前推荐的兼容方式
#### A. 调度层负责顺序
调度层负责表达：
- session 在 microcycle 中的顺序
- 休息日位置
- preserve order / preserve calendar / smart merge / manual 等恢复策略

#### B. 进步层不写死在周安排名称上
进步逻辑应避免出现如下错误设计：
- “如果是四分化就……”
- “如果是五分化就……”
- “如果每周固定练 5 次就……”

正确做法应是：
- 基于模板动作槽位
- 基于动作角色
- 基于策略类型
- 基于动作轨状态
- 基于本次是否发生有效暴露

#### C. 计划生成层只合成结果
计划生成层不需要理解“这是腿日还是上半身日的传统名称意义”，  
它只需要根据：
- 当前 session template 中有哪些动作槽位
- 当前槽位绑定什么 progression policy
- 当前 ProgressTrack 状态是什么

来合成这次 PlannedUnit 的目标安排与变化快照。

### 39.5 对动作轨的当前建议
当前默认建议仍应保持：

> 一个模板动作槽位 = 一条独立动作轨

这意味着即使同一个动作名在不同 session 中重复出现，例如：
- 胸日里的卧推
- 上半身日里的卧推

也不应默认共用同一条 ProgressTrack。  
因为它们可能在：

- 训练角色
- 容量目标
- 策略类型
- 进步节奏

上都不同。

只有未来明确需要时，才考虑引入“共享动作轨”机制。

### 39.6 当前对主项 / 辅项选择器的提醒
在混合型周安排中，“本次谁发生变化”的选择器不应是全局写死的。  
更合理的方向是：

- 先在当前 session 内筛选 eligible 槽位
- 再按角色与优先级选择本次变化对象

这意味着：
- 主项通常按自身脚本或线性规则推进
- 次主项与辅助项可进入当前 session 的变化池
- 选择器是 session-aware 的，而不是 split-name-aware 的

### 39.7 当前结论
对于如下类型的训练安排：

- 四分化
- 五分化
- 混合型周结构
- 固定周几但包含组合日
- 不规律轮转但保持顺序

系统的通用进步逻辑都应兼容。  
兼容的前提不是“提前为每种分化写专门逻辑”，而是：

> 坚持“调度层管结构，槽位层管策略，动作轨管状态，执行层管反馈”的分层设计。

### 39.8 当前阶段提醒
后续实现任何具体进步策略时，都不应假定：
- 固定周结构
- 固定分化名称
- 固定每周暴露次数

应优先基于：
- 动作轨的有效暴露
- 当前槽位配置
- 当前执行结果
- 当前恢复策略
来运行。

## 42. 逾期待处理与当日训练的“忽略 / 跳过 / 顺延”分流规则

### 42.1 目标
在“顺序优先、单队列、级联顺延”已成立的前提下，进一步把以下动作的业务语义分开：

- 忽略此次训练
- 跳过本次训练（不练了）
- 顺延本次训练（后续整体往后）

目标是：
- 不让“跳过”和“顺延”混成一类
- 不让每一次不练都触发整体重排
- 保持 progression 顺序可靠
- 让用户能按真实训练习惯处理逾期和当天训练

### 42.2 三类动作的严格语义
#### A. 忽略此次训练（用于逾期待处理）
语义：
- 当前最早未解决逾期训练，不再补练
- 该训练以“未训练 / skipped”结果归档
- 不消费有效 exposure
- 不触发后续未冻结队列整体顺延
- 当前边界推进到下一条未解决训练

本质：
> 这是“放弃补这次”，不是“把这次往后搬”。

#### B. 跳过本次训练 - 不练了（用于当天训练）
语义：
- 当前当天训练直接以“未训练 / skipped”结果归档
- 相当于把这次训练从当前未来队列中移除
- 后续训练日期保持正常，不整体后移
- 下一训练日按原计划继续

本质：
> 这是“删掉这次槽位”，不是“延期”。

#### C. 顺延本次训练（用于当天训练或逾期训练）
语义：
- 当前这条训练仍然要练，只是不在当前日期执行
- 当前这条从边界点开始重排到新的合法训练日期
- 后续所有未冻结训练整体顺延
- 顺序不变、单队列不变、同日唯一不变

本质：
> 这是“保留这次训练，并把后续整体往后推”。

### 42.3 当前推荐规则
#### A. 对最早逾期训练开放的动作
当前最早未解决训练，允许：
- 今天补练
- 改到某天（顺延后续）
- 忽略此次训练（未训练归档，不顺延后续）

#### B. 对当天训练开放的动作
当前当天训练，允许：
- 进入执行录入
- 跳过本次训练
  - 不练了（未训练归档，不顺延）
  - 顺延（整体后移）

### 42.4 对 progression 的影响
#### A. 忽略 / 不练了
- 当前训练记为 skipped / 未训练
- 不作为成功暴露
- 不触发正常推进
- 不应误判为部分完成
- 未来是否产生 hold / suggestion，可由后续规则决定，但当前最小逻辑是不消费成功暴露

#### B. 顺延
- 当前训练尚未落定结果
- 不立即推进动作轨
- 真正执行后才形成 exposure
- 后续计划先按顺延后的队列重排

### 42.5 对单队列原则的补充
引入“忽略此次训练 / 不练了”后，仍必须满足：

1. 同一活动计划下，未冻结 planned sessions 按 sequence_index 严格递增
2. 同一活动计划下，未冻结 planned sessions 的 session_date 严格递增
3. 同一活动计划下，每天最多一条 planned session
4. 忽略 / 不练了 只是冻结当前 session，不重排后续
5. 顺延才会触发级联重排

### 42.6 当前推荐实现方向
后续实现时，建议把“忽略 / 跳过 / 顺延”视为三种明确 action，而不是只用一个“skip”兜底：

- `archive_untrained`
- `skip_without_reschedule`
- `reschedule_with_cascade`

即使底层最终仍落到现有状态字段，也要在 use-case 层保持这三种业务意图的分离。

### 42.7 当前结论
到这一层为止，训练系统中的队列处理语义应明确为：

- 不补这次：归档未训练，队列前进
- 这次不练：归档未训练，后续正常
- 这次改天练：当前保留，后续整体顺延

这三者不可再混淆。
